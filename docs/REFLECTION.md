# V3 运单全流程管理系统 — 架构反思

本文档针对 V3 考核说明中的 6 个反思问题作答。所有回答均基于本系统真实落地的实现：Next.js 16 App Router + TypeScript、独立 Neon 数据库（autumn-waterfall-84758359）、通过 V2 只读 `/api/external/*`（API Key 鉴权 + X-Request-Id 链路追踪）拉取运单数据、复用 V2 的 JingTian 设计系统。两张状态机（工单物流态、扫描批次质检态）通过 `ticket_id` 解耦关联；一致性由数据库事务 + 乐观锁 + 幂等键共同保障；规则全部走 `system_config` / `qc_rules` 配置表，不存在硬编码。

---

## 1. 如果题目直接给出了精确的审批金额阈值和超时时长，你的架构设计会有什么不同？

在本系统的实际实现里，金额阈值（`L1_THRESHOLD=500`）和超时时长（`APPROVAL_TIMEOUT_HOURS=24`、`QC_HOLD_TIMEOUT_HOURS=2`）并没有写在代码里，而是落在 `system_config` 表中，运行时可配。如果题目把这几个数值直接写死成需求常量，最直接的变化是：我会把状态机里的 `amount > L1_THRESHOLD` 分支判断从"读 config"退化为"读常量"，省掉一次配置查询和配置缓存层（当前我用应用内缓存 + 写入时失效的策略来避免每次审批都打 config 表）。

但更深的架构差异在于：当阈值是"已知常量"时，L1 与 L2 的分流逻辑在编译期就能确定，理论上可以把工单按"是否会进 L2"拆成两条独立的处理流（fast-path / slow-path），分别走不同的索引、不同的审批队列，甚至不同的表——查询和并发模型都能针对各自流量做优化。而我现在的实现是单表 + 运行时分支，统一承载 L1/L2 两态，好处是状态机闭环、跳级逻辑（扫描质检失败自动进 L2）天然统一，代价是 L2 这条低频路径混在主表里，无法做物理隔离。

更关键的是超时时长。如果是固定值，定时任务只需扫一张静态阈值表；而我现在支持"每个工单按场景动态读取超时阈值"，定时扫描逻辑必须每行带配置上下文判断，复杂度更高。换言之，题目给的常量越精确，我"通用规则引擎 + 配置驱动"这一层的超额设计就越显得过度工程；但考虑到实际业务里 `MAX_RESUBMIT=3`、QC 与物流超时阈值不同、未来很可能按客户/品类分档，配置化是更接近真实生产的选择，常量化只是让 V3 这次的代码更短一点而已。这一点我在 `system_config` 的设计说明里也写了注记。

## 2. 你认为本题里"审批状态变更"与"库存/赔付联动"之间的一致性，最大的风险点在哪里？如果不用数据库事务，你会怎么处理？

最大风险点不在状态变更本身，而在"审批通过 → 工单进入 EXECUTING → 写 `compensation_records`（赔付记录，方向 = SUPPLIER/CUSTOMER）→ 写 `inventory_changes`（库存增减）→ 解锁扫描批次（scan_batches.status = UNLOCKED）"这条多步链路中任何一步失败后留下的"半执行"状态。具体来说有两类高危场景：

第一类是**重复执行**。审批接口如果没有幂等保护，前端重试或网络重发会导致同一笔赔付被写两次——尤其是向供应商追偿（QC 路径，payment_direction=SUPPLIER）这种直接涉及资金的写入。我的实现用 `Idempotency-Key` 请求头 + `approval_records.idempotency_key` 唯一约束来兜底，重复请求命中唯一约束后直接返回已有结果。第二类是**部分提交**。比如 `inventory_changes` 写成功但 `compensation_records` 回滚失败，或者批次解锁在赔付落库之前——就会出现"库存已经动、赔付没记账"或"批次放行了但赔付还没发起"的不一致，且事后无法逆向追溯。我通过 `compensation_records.approval_id` 和 `inventory_changes.approval_id` 同时外键到 `approval_records`，保证了反向可追溯；但这只有在事务包裹下才有意义——如果中间某步已经 COMMIT，外键也只是"事后拼尸检"，不能挽回数据正确性。

如果**不用数据库事务**，我会采用 saga + 本地补偿表的模式：把整个联动拆成若干个幂等的小步骤，每步完成后向一张 `task_outbox` 表写入一条"已执行"记录；任何一步失败时，按 `task_outbox` 反向回放补偿动作（比如已写的 `inventory_changes` 反向冲账）。但这套方案的代价远大于事务：需要为每个步骤实现正向 + 反向两套逻辑，补偿动作本身也要幂等，还要处理"补偿过程中宕机"的极端情况。因此在本题这种"单库、跨表、强一致"的场景下，我用 `BEGIN/COMMIT/ROLLBACK` 包裹审批联动是性价比最高的选择，乐观锁（`version` 字段冲突返回 409）则负责处理并发审批的竞态——这两者一起把"状态变更 + 联动"收紧成原子的临界区，比 saga 简单一个量级。

## 3. 面对一句话级别的模糊需求，你倾向于"先问清楚再动手"还是"先做出一个版本再迭代"？为什么？

在这个项目里，我的实际倾向是**"先基于 V2 既有约定做出一个可运行版本，同时把假设显式标注出来，再迭代"**，而不是"先问清楚再动手"。原因是 V3 的题目本身就嵌在一个明确的工程上下文里——V2 已经存在、运单结构已知、JingTian 设计系统已经成型、审批/赔付/质检的业务语义在物流行业是高度标准化的。这意味着一句话级别的模糊（比如"做异常工单管理"）背后，90% 的细节其实已经被 V2 的实现和行业惯例约束住了，真正需要澄清的只有 10% 左右（比如金额阈值的具体数值、超时具体几小时、误判放行要不要审计日志）。

更重要的是，"先做出一个版本"在本题里有具体的工程价值：我把所有不确定的数值都放进了 `system_config`（`L1_THRESHOLD`、`APPROVAL_TIMEOUT_HOURS`、`MAX_RESUBMIT` 等）和 `qc_rules`（条件用 JSONB 表达），这等于把"模糊"局部化到了配置层。如果产品经理后来改口"阈值不是 500 是 800"或"L1 超时不是 24h 是 12h"，我**完全不用改代码**，改一行 config 即可——这才是真正消化了"模糊需求"的成本。反过来，如果一开始就追问产品经理每个细节，反而会陷入"在没有可运行系统的情况下空谈参数"的低效讨论：产品经理在没有看到系统形态之前，往往自己也答不准这些数值。

当然这种策略有前提：假设必须被显式记录（我在 `system_config` 的每个 key 上都写了说明和默认值），并且可运行版本必须足够小，能在被推翻时低成本重做。V3 这种 CRUD + 状态机 + 配置驱动的系统正好满足这个前提，所以"先做版本再迭代"是更优解。

## 4. 如果产品经理后续要求新增第六种异常类型（比如"客户投诉服务态度"，不涉及货品本身、不涉及库存联动），你现在的表结构和状态机需要改动哪些地方？改动成本大不大？

改动成本**很小**，这正是当前架构的核心收益之一。具体改动点：

**表结构层面，几乎零改动**。当前异常类型用一个受控枚举（DAMAGED / LOST / WRONG / QUALITY_FAIL / OTHER 之类）+ `tickets.exception_type` 字段承载，新增"COMPLAINT_SERVICE"只需在枚举里加一个值。因为"服务态度投诉"不涉及库存联动，我**不需要**给它写 `inventory_changes` 记录，也**不需要**触发扫描批次状态机——`compensation_records` 的 `approval_id` 外键、`inventory_changes.approval_id` 外键都是 nullable / 可选写入的，所以"无库存联动"的异常类型天然能落在同一张 `tickets` + `approval_records` 表里，只是少写几张关联表。QC 相关字段（`qc_hold`、`scan_batch_id`）留空即可。

**状态机层面，零改动**。服务态度投诉走的是纯物流态（PENDING → L1_REVIEWING → [可选 L2] → EXECUTING → COMPLETED），这正是工单状态机已经覆盖的主路径。它**不进入**扫描批次状态机（没有 scan、没有 QC 规则引擎、没有 LOCKED/UNLOCKED），所以扫描状态机那一边一行代码都不用动。唯一需要新增的是**赔付方向**：服务态度投诉的理赔对象是客户（payment_direction=CUSTOMER），这和现有的物流理赔客户走同一条路径，无需新枚举。

**真正需要写的新代码只有三块**：(1) 异常类型枚举加一个值 + 列表筛选下拉加一个选项；(2) 如果这类投诉有专属的处置动作（比如"道歉回访""补偿优惠券"而非"退货/补货/降级"），需要给它配一套处置流程模板——但因为我已经把 QC 的处置动作（RELEASE/RETURN_SUPPLIER/REPURCHASE/DOWNGRADE）和物流的处置逻辑解耦，新增"服务态度"专属动作只需要在它的处置枚举里加值，不影响其他类型；(3) 如果产品希望这类投诉走更轻的审批（比如金额低于某阈值直接 COMPLETED 跳过 L2），在 `system_config` 里加一个 key 即可。综合下来，这属于"半天工作量"的扩展，状态机和一致性机制完全复用，这正是把两张状态机解耦、把规则配置化带来的红利。

## 5. 如果工单量从 200 条增长到 20 万条，你的系统里最先撑不住的环节是哪个（列表查询？定时任务全表扫描？接口同步积压？）？你会怎么处理？

最先撑不住的会是**定时任务全表扫描**，其次是**列表查询的复合筛选**，接口同步积压（V2 调用）反而相对扛得住。逐个分析：

**定时任务（最先撑不住）**。当前超时升级（L1 超时 → L2、L2 超时 → 自动驳回）和 QC 驻留超时（QC_HOLD_TIMEOUT_HOURS=2）靠定时任务扫表。200 条时全表扫描毫无压力，但 20 万条时如果任务每几分钟扫一次"状态 = 审批中 AND created_at < now - timeout"的全表，会产生持续的慢查询和锁竞争，尤其在 Neon 这种共享存储上会拖累其他查询。处理方式：(1) 给 `tickets(status, created_at)`、`scan_batches(status, created_at)` 建复合索引，让扫描走索引范围扫而非全表；(2) 把"超时判断"从"扫全表算时间差"改为"延迟队列"——审批进入时计算好到期时间，写入一张 `timeout_queue(deadline, ticket_id)` 表，定时任务只扫 `deadline < now` 的行，扫描量从"全表活跃工单"降到"恰好到期的少数"，量级下降两到三个数量级；(3) 超时扫描切片化，每次只处理一个 `ticket_id` 范围段，避免长事务。

**列表查询**。当前工单列表的分页 + 多条件筛选（状态、异常类型、金额区间、时间区间、客户）在 20 万行时如果缺少合适的复合索引，`COUNT`（用于分页总数）会率先变慢。处理方式：为高频筛选组合建覆盖索引，把 `COUNT` 换成估算或游标分页（keyset pagination，按 `created_at DESC, id DESC` 翻页，避开 `OFFSET`），并在 V2 数据快照（`waybill_snapshots` 表）上冗余关键字段，避免每次列表都跨服务拉 V2。

**V2 接口同步积压（相对扛得住）**。当前 `v2-client.ts` 有 8s 超时 + 1 次重试 + 失败降级到 `waybill_snapshots` 快照，所以即便 V2 偶发慢，V3 自己的读写不会阻塞——快照兜底保证了 V3 的可用性。真正的风险在于"快照陈旧"：如果 V2 长时间不可用，`api_sync_logs` 里会积累大量降级标记（data-source = CACHE），用户体验上看到的是旧数据。处理方式是给快照加 TTL 标注 + 在 UI 上显式提示"数据来源：缓存（更新于 X 分钟前）"，并增加同步队列（批量 + 限流）防止恢复期瞬时打爆 V2。

## 6. 你在给 V2 新增对外接口时，如何保证不破坏 V2 现有的调用方？如果 V2 的运单金额字段类型从 int 改成 decimal，你会如何处理 V3 侧的兼容？

**新增对外接口不破坏现有调用方**，我的做法分四层。第一层，**路径隔离**：新增的对外接口全部收口在 `/api/external/*` 这个独立前缀下，与 V2 既有的内部 API 路径（比如 `/api/waybills/*`）物理分离，老调用方不会因为路由变化而受影响。第二层，**鉴权隔离**：`/api/external/*` 强制走 API Key 中间件 + X-Request-Id 链路追踪，而 V2 内部接口保持原有鉴权方式不变，两套机制的演进互不耦合。第三层，**版本化**：对外接口在 URL 或 header 里带版本（`/api/external/v1/...`），未来字段变更走 `v2`，老版本保留一个废弃周期，让 V3 有时间平滑迁移。第四层，**契约测试 + 监控**：用 `api_sync_logs` 记录每次外部调用的 request_id、响应码、error_class，并对接一套契约测试用例固定 V2 字段结构，一旦 V2 内部改动意外波及 `/api/external/*`，测试和线上监控会同时告警。

**V2 运单金额从 int 改成 decimal**，在 V3 侧我会这样兼容。首先，V3 侧的 TypeScript 类型从一开始就不应该写死成 `number`——但即便现在写死了，处理方式也是：(1) 在 `v2-client.ts` 的响应解析层做一次类型归一化，把 V2 返回的 decimal（可能是字符串 `"123.45"`，也可能是数字 `123.45`）统一 `parse` 成 V3 内部的金额表示（建议直接用"分为单位"的整数 `12345`，避免浮点精度问题；如果业务需要小数，用 `decimal.js` 这类库而非原生 `number`）。(2) V3 自身数据库（Neon）里 `waybill_snapshots`、`tickets`、`compensation_records` 的金额列，因为是我自己建的表，应当一开始就用 `NUMERIC(12,2)` / `DECIMAL` 而非 `INT`，这样无论 V2 怎么变，V3 内部的金额存储始终是 decimal-safe 的——本题里"理赔金额"本身就需要小数（百分比定损、运费小数），所以 V3 侧用 decimal 是基本要求而非过度设计。(3) 在 `api_sync_logs` 里记录 V2 返回的原始 payload，方便在类型迁移期间对账：如果发现某条工单的金额在 V2 是 int、在 V3 解析后丢精度，能快速定位到具体的 request_id。(4) V3 业务逻辑（比如 `amount > L1_THRESHOLD=500` 的 L1/L2 分流）的阈值判断，必须用 decimal-safe 的比较，不能用 `===` 直接比浮点数——这一点在状态机分支里要特别检查。整体上，因为 V2 对 V3 是"只读数据源"而非"共享数据库"，V3 有完整的解析层做适配，V2 字段类型变更的影响可以被限制在 `v2-client.ts` 这一个文件里，不会扩散到业务逻辑。
